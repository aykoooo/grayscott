## 📐 Key Technical Details

### Current Architecture
- **Workgroup**: 16×4 (64 threads) — reshaped for warp locality
- **STRIDE**: TX+2 = 18 (tile width includes 1-cell halo each side)
- **Tile**: 18×6 shared memory arrays for u/v
- **Command buffer batching**: All dispatches in one submit — DONE, sacred
- **Ping-pong**: U0↔U1, V0↔V1 with two bind groups
- **Periodic boundaries**: `select()` for both tile loading and Laplacian
- **Pearson variant**: Neumann boundaries (`max`/`min`) + spatial feed/kill maps
- **Hash**: `e16ed0e3c29cc50b5fa2b42791f31ab00b39d488e971b5d3c6017970ed037a43` — sacred

### Performance
- Peak (native RTX 4060): 2.35B cells/sec (8×8 tiling + batched dispatch)
- Recent best: ~681M cells/sec (16×4 reshape, on thermally-degraded session)
- Peak (step-only at 1024²): ~4.2B cells/sec
- CPU reference: ~500M cells/sec (single-threaded ReleaseFast)
- Theoretical BW ceiling: 13.25B cells/sec at 272 GB/s
- Current utilization: 17.7% of bandwidth ceiling

### Browser Subgroup Layout (16×4)
```
Warp 0: threads 0..31 = rows 0 and 1 (both 16-wide)
Warp 1: threads 32..63 = rows 2 and 3
```
- Thread (row y, col x): lane_id = y*16 + x within warp half
- Horizontal ±1 neighbors: lanes ±1 in same 16-thread row
- Vertical ±1 row neighbor: lanes ±16 (within same 64-thread warp across 2 rows)

### Browser Benchmark Architecture (Phase 11)
- **Seed generation**: `gs_wasm_generate_seeds(w,h)` uses exact same `std.Random.DefaultPrng.init(42)` as native — guarantees hash match
- **WASM memory model**: Exports return structs `{ptr, len}` pointing into WASM linear memory; JS reads via `new Uint8Array(memory.buffer, ptr, len)`
- **Timing**: `performance.mark` → `queue.submit()` → `await queue.onSubmittedWorkDone()` → `performance.measure`. Never use naive wall-clock (submit is async).
- **Hash**: `crypto.subtle.digest("SHA-256", mappedArrayBuffer)` → hex string
- **Serving**: MUST be localhost (WebGPU secure context). `.wasm` needs `application/wasm` MIME. Use `npx serve benchmark/`.
- **Feature detection**: `adapter.features.has("subgroups")`, `.has("shader-f16")` — both optional, graceful fallback
- **Pure compute**: No `<canvas>` required. Just request adapter + device.
- **Bind group layout**: 5 entries matching PERIODIC_LAYOUT (bindings 0-3 = storage, 4 = uniform, 20-byte params struct)
- **All 8 stencil neighbors accessible via subgroup shuffles within single warp**

| Browser | Features | Shader variant |
|---|---|---|
| Chrome 134+ | subgroups, shader-f16 | generateWgslSubgroups() |
| Firefox/Safari | none yet | generateWgsl() (standard tiling) |

## 🧪 Already Explored & Exhausted

- **f16 storage (Phase K)**: Previously reverted due to "compute-bound" diagnosis — this was WRONG.
  Roofline model proves bandwidth-bound (AI=1.56 FLOPs/byte at 272GB/s, ridge=55.6).
  Phase 12 REVISITS this as highest-impact remaining optimization. SMEM reads are the bottleneck;
  f16 halves SMEM traffic. ~+40-80% expected.
- **vec2 UV global packing**: -38%, register pressure > coalescing benefit
- **Bank conflict padding**: <5% impact at 8×8 workgroup, below noise floor
- **Temporal blocking (naive)**: Requires dual SMEM tiles + coarsening groundwork; complex on 64 threads.
  Phase 15 revisits after f16/coarsening results are in.
- **Diamond/wavefront tiling**: Overkill for 9pt 2D stencil
- **Tensor cores**: FP32 scalar stencil doesn't map to matrix multiply
- **Semi-stencil**: Only benefits r≥3 (ours is r=1)
- **Multi-GPU / async copy / persistent threads**: Not applicable in wgpu/browser
- **Thread coarsening with global reads**: -34% (cell B used global memory). Correct approach uses SMEM expansion (Phase 14).
